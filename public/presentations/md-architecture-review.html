Current Architecture (Logical)

Presentation Layer

Next.js App Router pages + reusable layout components (header, top strip, footer, menus).

Interaction Layer

Client components for maps, accessibility tools, embeds, form flows, and dynamic controls.

Domain Services

`services/*` encapsulates validation, ingestion, workflow logic, and persistence contracts.

Data Layer

PostgreSQL pooled access; fallback defaults where DB is unavailable for non-critical reads.

Data & Integration Design

  • PostgreSQL-backed content and workflow tables support governed updates and audit traceability.
  • External procurement sources (GeM/CPP) integrated with parser + fallback strategy.
  • State project module now supports API submission and count aggregation model.
  • Service contracts isolate external source volatility from page rendering logic.

Ingestion

Periodic fetch with revalidation and parser controls.

Validation

Server-side checks before persistence for data quality.

Fallback

Graceful content defaults to preserve uptime perception.

Data ReliabilitySlide 5

Security & Governance Posture

  • Workflow APIs enforce role checks for proposal, approval, and rejection pathways.
  • Input validation implemented in service layer for forms and API submissions.
  • Authentication route exists for role-aware portal entry and controlled redirects.
  • Governance model already aligns to auditable publish lifecycle.
Note: header-based role simulation should be upgraded to token/session-backed enforcement for production hardening.

Public InterfaceSlide 7

Operational Risks (Current)

Parser Drift

External page markup changes can affect GeM/CPP data extraction.

Auth Hardening Gap

Role simulation pattern must move to stronger identity controls.

Observability Gap

Central dashboards for API SLAs and parser health not yet institutionalized.

Data Maturity

Certain modules still depend on staged/demo baselines until full data onboarding.
